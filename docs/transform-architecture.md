# 변환 엔진 구현 방향

## 목적

알고리즘 변환기는 UI와 변환 규칙을 분리해서 구현한다. 화면 코드는 입력, 퍼즐 선택, 출력 표시, 복사 같은 사용자 인터페이스만 담당하고, 실제 변환 규칙은 별도 변환 엔진에서 처리한다.

## 기본 구조

변환 로직은 `src/transforms/` 아래에 둔다.

```text
src/
  main.js
  styles.css
  transforms/
    index.js
    3x3.js
    square1.js
    fto.js
```

각 파일의 역할은 다음과 같다.

- `main.js`: 화면 상태 관리, 퍼즐 선택, 입력값 읽기, 출력 카드 렌더링, 복사 버튼 처리
- `transforms/index.js`: 퍼즐 ID에 맞는 변환기 선택, 공통 `transformAlgorithm(puzzle, algorithm)` 제공
- `transforms/3x3.js`: 3x3 파싱과 역공식, 좌우대칭, 전후대칭, 상하대칭 변환
- `transforms/square1.js`: Square-1 파싱과 역공식, 전후대칭, 상하대칭 변환
- `transforms/fto.js`: FTO 파싱과 역공식, 좌우대칭, 전후대칭 변환

## 처리 흐름

변환은 다음 순서로 처리한다.

```text
입력 문자열
  -> 토큰 파싱
  -> 퍼즐별 변환
  -> 출력 문자열 포맷팅
```

임시 문자열 치환보다 명시적인 토큰 기반 처리를 우선한다. 문법 파싱과 퍼즐 상태의 합법성 검사는 분리한다.

## 출력 인터페이스

UI는 변환 결과를 다음 형태로 받는다.

```js
{
  inverse: "",
  mirrorLeftRight: "",
  mirrorLeftRightInverse: "",
  mirrorFrontBack: "",
  mirrorFrontBackInverse: "",
  mirrorUpDown: "",
  mirrorUpDownInverse: ""
}
```

퍼즐에서 지원하지 않는 변환은 반환하지 않거나 빈 문자열로 둔다. UI는 퍼즐별 출력 카드 목록을 기준으로 표시 여부를 결정한다.

`대칭의 역공식`은 원본 공식의 역공식을 대칭하는 것이 아니라, 대칭 변환 결과를 다시 역공식으로 변환한 값이다.

```text
좌우대칭의 역공식 = inverse(mirrorLeftRight(input))
전후대칭의 역공식 = inverse(mirrorFrontBack(input))
상하대칭의 역공식 = inverse(mirrorUpDown(input))
```

대칭 출력은 페이지 안에서 1개 컬럼 카드 목록으로 보여준다. `역공식`은 독립 카드로 유지하고, 각 대칭 카드는 대칭 공식과 대칭의 역공식을 같은 카드 안에 세로로 표시한다.

| 카드 | 카드 안의 출력 |
|---|---|
| 좌우대칭 | 좌우대칭, 좌우대칭의 역공식 |
| 전후대칭 | 전후대칭, 전후대칭의 역공식 |
| 상하대칭 | 상하대칭, 상하대칭의 역공식 |

`역공식`은 원본 공식에 대한 독립 출력으로 유지한다.

입력에 줄바꿈이 있으면 출력도 같은 줄바꿈 구조를 유지한다. 각 줄은 독립적으로 변환한다.

한 줄에서 `//`가 처음 나온 지점부터 줄 끝까지는 라인 주석으로 처리한다. 라인 주석은 변환하지 않고 출력에 그대로 붙인다. 여러 줄 주석은 지원하지 않는다.

```text
입력: R U R' U' // 트위스트
역공식: U R U' R' // 트위스트
```

3x3과 FTO의 공식 부분에서는 가독성용 구분자인 `(`, `)`, `,`, `/`를 공백처럼 취급한다. 소괄호 짝 검사는 하지 않는다. `//`는 라인 주석으로 먼저 분리한 뒤 공식 부분에만 이 규칙을 적용한다.

```text
입력: R U / F B / ( R U R' F)
해석: R U F B R U R' F
```

## 확장 계획

커뮤테이터와 컨쥬게이트 축약 입력은 라인 주석 분리 후, 일반 토큰 파싱 전에 확장한다.

- 커뮤테이터 `[X,Y]`: `X Y X' Y'`로 확장한다.
- 컨쥬게이트 `X:Y`: `X Y X'`로 확장한다.
- `X`, `Y`는 단일 토큰이 아니라 공식 덩어리일 수 있다.
- 중첩 표현을 허용한다. 예를 들어 `R : [D R, U F]`처럼 컨쥬게이트 안에 커뮤테이터를 넣을 수 있다.
- 3x3에 먼저 적용한다. FTO도 같은 문법을 사용할 예정이지만, 현재 FTO 변환기는 준비 중이다.
- Square-1은 `/`가 실제 표기라 별도 설계 후 적용한다.

예상 해석은 다음과 같다.

```text
[ R, U F]       -> R U F R' F' U'
R : U F         -> R U F R'
R : [D R, U F]  -> R D R U F R' D' F' U' R'
```

## 구현 원칙

- UI와 변환 규칙을 같은 파일에 섞지 않는다.
- 퍼즐별 문법과 변환 규칙은 파일을 분리한다.
- 파서, 변환기, 포맷터를 가능한 한 순수 함수로 작성한다.
- 나중에 테스트를 붙일 수 있도록 DOM에 의존하지 않는 함수로 구현한다.
- 3x3, Square-1, FTO 표기 판단은 로컬 표기법 스킬의 기준을 따른다.
